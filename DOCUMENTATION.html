Introduction

HDM School & Cyber is a comprehensive, dual‑system platform designed to manage the operations of a computer school and a cyber café business. Built with the MERN stack (MongoDB, Express.js, React, Node.js), it offers robust, secure, and user‑friendly interfaces for both systems, each with its own database and authentication.

The School Management System handles student enrollment, employee records, fee collection, inventory (computers/printers), accounts, a student/staff portal, and an online application process with admission letter generation. The Cyber Services System tracks service income, expenses, inventory, and generates detailed reports.

This document provides a detailed overview of the system's architecture, features, technology stack, API endpoints, deployment procedures, and future roadmap. It serves as both a technical reference for developers and a project summary for stakeholders.

1. Features

School Management System

  • Landing Page – Dynamic content (school name, motto, courses, contact) from settings.
  • Student Enrollment – Registration number auto‑generated, computer assignment from inventory, completion date auto‑calculated, initial fee payment recorded.
  • Employee Management – Auto‑generated employee IDs (EM‑001, EM‑002…), salary payment recording.
  • Fee Management – Record payments, print receipts, generate fee reports, view payment history per student.
  • Inventory Management – Track computers, printers, furniture; computers auto‑synced from settings (range or manual).
  • Accounts – Income/expense tracking, balance, transaction history with print vouchers.
  • Student/Staff Portal – Register using registration number (from student/employee records), login, view profile, fee status, payment history, print receipts.
  • Online Applications – Landing page application form, admin dashboard to review, accept/reject, generate admission letter (editable message, stamp, watermark).
  • Settings – Configure school name, motto, address, stamp image, courses (with description), computer list (range or manual with values).
  • Print Functionality – All lists and individual records have print buttons with school header, stamp, watermark, and footer (school name, address, print time, motto).
  • Role‑Based Access – Admin (full) and staff (limited) roles.

Cyber Services System

  • Dashboard – Real‑time overview: today's income, current balance, inventory value.
  • Service Recording – Record services (printing, photocopy, internet, etc.) with auto‑calculated amount based on quantity and price.
  • Inventory Management – Track cyber assets (computers, printers, scanners) with values.
  • Accounts & Expenses – Add expenses (e.g., electricity, internet), view transaction history, print expense vouchers.
  • Reports – Generate general, services, expenses, financial, or inventory reports with date filters; print with watermark and footer.
  • Settings – Configure business name, address, contact, stamp image, services list (with optional default price), tax rate, receipt footer text.
  • Print Receipts – Service receipts and expense vouchers include business header, stamp, and footer.

2. Technology Stack

Frontend (React + Vite)

  • React 18 – Functional components, hooks, and context API.
  • React Router v6 – Client‑side routing (separate routes for landing, admin, portal).
  • Axios – HTTP client with interceptors for token management.
  • Tailwind CSS – Modern, responsive styling with custom blue/green theme.
  • html2pdf.js – Direct PDF download for receipts and reports.
  • Font Awesome – Icons for sidebar and buttons.

Backend (Node.js + Express)

  • Node.js – JavaScript runtime.
  • Express.js – RESTful API framework with sub‑app routing for School and Cyber.
  • MongoDB + Mongoose – Two separate databases (school_db and cyber_db).
  • JWT – Authentication and authorization (separate secrets for School and Cyber).
  • bcryptjs – Password hashing.
  • CORS – Configured for localhost and production domains.
  • dotenv – Environment variable management.

3. System Architecture

The system follows a client‑server architecture with two frontends and a single backend serving both:

  • School Frontend (React): Served at port 3000. Contains public landing page, admin portal (sidebar), and student/staff portal. All dynamic data fetched from backend /api/school/* endpoints.
  • Cyber Frontend (React): Served at port 3001. Contains admin dashboard with sidebar. Fetches data from /api/cyber/* endpoints.
  • Backend (Node.js/Express): Exposes two independent sub‑apps (/api/school and /api/cyber), each with its own database connection and JWT secret. All business logic, authentication, and database interactions are handled here.
  • Databases (MongoDB): Two separate databases – school_db and cyber_db – ensuring complete data isolation.

Communication is stateless; each request includes a JWT token. Admin users for School and Cyber are stored in separate collections and cannot cross‑authenticate.

4. Database Schema (MongoDB Collections)

School Database (comp_school)

  • school_users – Admin/staff accounts.
  • students – regNumber, name, age, gender, phone, idNumber, enrollmentDate, completionDate, course, feesPaid, computerAssigned.
  • employees – empId (auto‑generated), name, age, gender, phone, idNumber, duty, salary, paymentMethod, etc.
  • fees – regNumber, studentName, amount, balanceAfter, date, notes.
  • inventory – name, type, value, status, assignedTo, assignedModel, purchaseDate, serialNumber, notes.
  • portal_users – regNumber, name, email, passwordHash, role, userId, active.
  • settings – schoolName, motto, address, phone, email, stampImage, courses (with description), computers (range/manual), landing page content.
  • applications – name, email, phone, course, message, status, admissionLetterGenerated, appliedDate.
  • transactions – type (in/out), amount, description, reference, date.

Cyber Database (cyber)

  • cyber_users – Admin accounts.
  • services – serviceType, amount, dateTime, description.
  • transactions – type (in/out), amount, description, reference, date.
  • inventory – name, type, value, status, purchaseDate, serialNumber, notes.
  • settings – businessName, address, phone, email, stampImage, services (with price), taxRate, receiptFooterText.

6. Security Considerations

  • Authentication: JWT tokens with separate secrets for School and Cyber; passwords hashed with bcrypt.
  • Authorization: Role‑based middleware protects admin routes.
  • Database Isolation: Two separate MongoDB databases prevent data cross‑access.
  • CORS: Configured to allow only trusted domains (localhost, production URLs).
  • Environment Variables: Sensitive keys stored in .env, never committed.
  • HTTPS: All traffic encrypted in production via SSL.

7. Deployment Guide

Backend

  1. Set NODE_ENV=production and configure environment variables in .env.
  2. Install dependencies: npm install --production.
  3. Use process manager: pm2 start server.js --name hdm-backend.
  4. Set up reverse proxy with Nginx to forward /api/* to the backend port (5000).
  5. Obtain SSL certificate via Let's Encrypt.

Frontend (School)

  1. Set VITE_API_BASE_URL=https://hdmserver.pxxl.click/api/school in .env.production.
  2. Build: npm run build.
  3. Upload dist/ folder to static hosting (e.g., Nginx, Vercel).

Frontend (Cyber)

  1. Set VITE_API_BASE_URL=https://hdmserver.pxxl.click/api/cyber in .env.production.
  2. Build: npm run build.
  3. Upload dist/ folder to static hosting.

All services are currently live at the URLs provided in the Developer Information section.
